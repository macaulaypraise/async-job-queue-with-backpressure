# Async Job Queue with Backpressure & Priority Scheduling

> A production-grade async job queue demonstrating backpressure, weighted
> fair scheduling, crash recovery, and dead letter queue patterns — built
> from scratch without Celery or Dramatiq to show the internals.

## What This Demonstrates

This project implements the mechanics that frameworks like Celery and SQS
hide behind abstractions: Redis Streams consumer groups for at-least-once
delivery, a two-watermark backpressure band to prevent queue overflow,
weighted fair scheduling to prevent priority starvation, and a visibility
timeout reaper for crash recovery. Every design decision is explained in
[DESIGN.md](./DESIGN.md).

## Architecture

```
Client → POST /jobs → Backpressure check → PostgreSQL (job state)
                                         → Redis Streams (3 priority queues)

Worker → Weighted scheduler (60/30/10) → XREADGROUP → Execute + heartbeat
       → Success: XACK + COMPLETED
       → Retryable failure: exponential backoff + re-enqueue
       → Permanent failure: Dead Letter Queue

Reaper (10s) → Per-priority visibility timeouts → Re-enqueue crashed workers
             → Zombie detection (stale heartbeat > 60s)
```

Full component diagram and decision log: [DESIGN.md](./DESIGN.md)

## Key Engineering Decisions

- **Redis Streams over Redis LIST** — XREADGROUP keeps messages in PENDING
  state until ACKed. A worker crash doesn't lose the job.
  → [Why in DESIGN.md](./DESIGN.md#1-redis-streams-over-a-simple-redis-list)

- **Two-watermark backpressure band** — Reject above HIGH_WATERMARK (10k),
  resume below LOW_WATERMARK (2k). Single thresholds oscillate.
  → [Why in DESIGN.md](./DESIGN.md#2-two-watermark-backpressure-band)

- **Weighted fair scheduling** — Critical/high/normal at 60/30/10.
  Strict priority causes starvation. Weights are runtime-configurable.
  → [Why in DESIGN.md](./DESIGN.md#3-weighted-fair-priority-scheduling)

- **Exponential backoff with full jitter** — Prevents thundering herd when
  many jobs fail simultaneously. `min(base * 2^n, 60s) * random(0,1)`.
  → [Why in DESIGN.md](./DESIGN.md#6-exponential-backoff-with-full-jitter-on-retry)

## Performance

Load tested with Locust at 50 concurrent users (legitimate submitters +
burst users + mixed priority workload) against a single-process dev server
on a WSL2 laptop:

| Metric | Value |
|---|---|
| Sustained throughput | ~56 req/s |
| P50 latency | 220ms |
| P95 latency | 640ms |
| P99 latency | 780ms |
| Backpressure trigger | 10,000 queued jobs → 503 + Retry-After |

> Note: dev server runs single-process (`uvicorn --reload`). Production
> deployments use Gunicorn with multiple Uvicorn workers.

## Quick Start

```bash
git clone <repo-url> && cd async-job-queue
make dev        # starts app + redis + postgres + prometheus + grafana
sleep 20        # wait for migrations to run
```

Submit a job:

```bash
curl -X POST http://localhost:8000/jobs \
  -H "Content-Type: application/json" \
  -d '{"payload": {"type": "send_email", "to": "you@example.com"}, "priority": "high"}'
```

Check its status:

```bash
curl http://localhost:8000/jobs/<job-id>
```

Check queue state:

```bash
curl http://localhost:8000/queues/metrics
```

## API Reference

Interactive docs auto-generated by FastAPI: **http://localhost:8000/docs**

| Endpoint | Description |
|---|---|
| `POST /jobs` | Submit a job (202 Accepted or 503 if backpressure active) |
| `GET /jobs/{id}` | Poll job status and result |
| `GET /queues/metrics` | Queue depths, DLQ depth, backpressure state |
| `PATCH /queues/weights` | Update scheduler weights at runtime |
| `GET /queues/weights` | Current scheduler weights and source |
| `DELETE /queues/weights` | Reset weights to config defaults |
| `POST /queues/dlq/{id}/replay` | Replay a dead letter queue message |
| `GET /health` | Health check for Redis and app |
| `GET /metrics` | Prometheus metrics |
